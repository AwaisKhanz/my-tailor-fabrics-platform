1. Architectural Decisions — All Gaps Resolved 
All 8 previously open questions have been answered. This section documents every decision so there is 
zero ambiguity when writing code. 
 
Decision 1 — Pricing Model 
RESOLVED: Two separate prices per GarmentType set by Admin: 
  • customerPrice  (paisas) — what the shop charges the customer per piece 
  • employeeRate   (paisas) — what the tailor earns per completed piece 
 
These are set globally per garment type. Branches can override both independently. 
When creating an order item, both prices are SNAPSHOT-copied onto the OrderItem row 
(unitPrice + employeeRate) so historical records are never affected by future price changes. 
 
Order Total = SUM(unitPrice × quantity) across all non-cancelled items − discountAmount 
Receipt shows: line items, customer prices, discount, total, advance paid, balance due. 
Employee earning = SUM(employeeRate × quantity) for COMPLETED/DELIVERED items only. 
Cancelled items → employee earns NOTHING (zero pay, no exceptions). 
 
Decision 2 — Branch Pricing 
RESOLVED: Global default + branch can override. 
 
GarmentType has: customerPrice (global default) + employeeRate (global default) 
BranchGarmentPriceOverride table allows per-branch overrides of either price. 
When creating an order item: check for branch override first, fall back to global. 
Admin panel shows both global prices and any branch-specific overrides in one screen. 
 
Decision 3 — Item Completion 
RESOLVED: All-or-nothing per line item. 
 
An OrderItem with quantity=3 is either PENDING, IN_PROGRESS, COMPLETED, or CANCELLED. 
Staff marks the entire line item as completed — no partial qty tracking. 
If a customer wants to split (e.g. take 2 now, 1 later), create 2 separate line items. 
This keeps the schema simple and the payment calculation unambiguous. 
 
Decision 4 — Overdue Orders 
RESOLVED: Auto-change status to OVERDUE + show alert. 
 
A scheduled cron job runs every hour (NestJS @Cron). 
Any order with dueDate < NOW() and status NOT IN (DELIVERED, COMPLETED, CANCELLED) 
is automatically transitioned to OVERDUE status. 
OVERDUE orders appear with a red badge in all list views. 
Dashboard shows an 'Overdue Orders' KPI card with count + quick-link to list.

Decision 5 — Cancelled Order Payment 
RESOLVED: No pay for cancelled items. Zero exceptions. 
 
When an order is CANCELLED, ALL its OrderItems are set to status = CANCELLED. 
Cancelled items are excluded from employee balance calculation at the DB query level. 
No manual override needed — the rule is absolute and enforced in the query. 
This is clearly communicated on the Payment screen: 'Earnings from completed items only.' 
 
Decision 6 — Order Discounts 
RESOLVED: Both fixed amount AND percentage discounts supported on an order. 
 
Order table has: discountType (FIXED | PERCENTAGE | null), discountValue (Int paisas or %). 
Only Admin+ roles can apply a discount. 
Calculation: 
  subtotal        = SUM(unitPrice × qty) for non-cancelled items 
  discountAmount  = FIXED: discountValue  |  PERCENTAGE: FLOOR(subtotal × discountValue / 
100) 
  totalAmount     = subtotal − discountAmount 
  balanceDue      = totalAmount − SUM(all OrderPayments) 
Receipt shows subtotal, discount line, and final total separately. 
 
Decision 7 — Employee Login 
RESOLVED: Employees get their own login (EMPLOYEE role). 
 
Role: EMPLOYEE — lowest privilege level, read-only, scoped to own data only. 
Employee login is linked to their Employee record via employeeId field on User. 
What EMPLOYEE role can see: 
  • Their own assigned order items (status, garment type, due date, description) 
  • Their own payment history and current balance 
  • Their own profile (read-only) 
What EMPLOYEE role CANNOT see: 
  • Other employees' data, payments, or orders 
  • Customer details (name, phone, address, measurements) 
  • Order pricing (unitPrice hidden — only their employeeRate shown) 
  • Any admin config or management screens 
Login via same /auth/login endpoint — role determines dashboard shown. 
 
Decision 8 — Database & Backups 
RESOLVED: Managed database (Supabase / Neon / Railway). 
 
Recommended: Neon (PostgreSQL-native, branching for dev/staging, free tier generous)

3. Complete Database Schema (Prisma) 
Critical Conventions 
1. MONEY: All monetary values stored as INTEGER (paisas). Rs 1,500.00 = 150000. 
   Never store floats for money. Divide by 100 for display. Use formatPKR() utility. 
2. SOFT DELETE: All main entities have deletedAt DateTime?. 
   ALWAYS add WHERE deletedAt IS NULL to queries. Never hard delete. 
3. BRANCH SCOPE: Every user-facing entity has branchId. 
   BranchGuard auto-applies this filter. Super Admin can override via X-Branch-Id header. 
4. SNAPSHOT PRICES: OrderItem stores unitPrice + employeeRate at time of creation. 
   This ensures price changes never affect historical orders. 
5. AUDIT: AuditInterceptor automatically logs all POST/PUT/PATCH/DELETE calls.

5.2 Order Total Recalculation 
// Called after any OrderItem is added, updated, or cancelled 
// Also called after discount is applied 
async recalcOrderTotals(orderId: string, tx?: PrismaTransaction) { 
  const db = tx ?? this.prisma; 
  const items = await db.orderItem.findMany({ 
    where: { orderId, status: { not: 'CANCELLED' } }, 
    select: { unitPrice: true, quantity: true }, 
  }); 
  const subtotal = items.reduce((sum, i) => sum + i.unitPrice * i.quantity, 0); 
 
  const order = await db.order.findUniqueOrThrow({ 
    where: { id: orderId }, 
    select: { discountType: true, discountValue: true, totalPaid: true }, 
  }); 
 
  let discountAmount = 0; 
  if (order.discountType === 'FIXED') { 
    discountAmount = Math.min(order.discountValue, subtotal); // never go negative 
  } else if (order.discountType === 'PERCENTAGE') { 
    discountAmount = Math.floor(subtotal * order.discountValue / 100); 
  } 
 
  const totalAmount = subtotal - discountAmount; 
  const balanceDue  = Math.max(0, totalAmount - order.totalPaid); 
 
  await db.order.update({ 
    where: { id: orderId }, 
    data: { subtotal, discountAmount, totalAmount, balanceDue }, 
  }); 
}

5.3 Employee Balance Calculation 
// src/payments/payment.service.ts 
async getEmployeeBalanceSummary(employeeId: string) { 
  const [earnedResult, totalPaid] = await Promise.all([ 
    // Raw SQL: SUM(employeeRate * quantity) for all COMPLETED/DELIVERED items 
    this.prisma.$queryRaw<[{ total: BigInt }]>` 
      SELECT COALESCE(SUM(employee_rate * quantity), 0) AS total 
      FROM order_items 
      WHERE employee_id = ${employeeId} 
        AND status IN ('COMPLETED', 'DELIVERED')`, 
    this.prisma.payment.aggregate({ 
      where: { employeeId }, 
      _sum: { amount: true }, 
    }), 
  ]); 
 
  const totalEarned = Number(earnedResult[0].total); 
  const paid        = totalPaid._sum.amount ?? 0; 
  const balance     = totalEarned - paid; 
 
  // Validate: balance should never go negative (guard in disbursement) 
  return { 
    totalEarned,          // all-time 
    totalPaid: paid,      // all-time 
    currentBalance: balance, 
    weekly: await this.getWeeklyBreakdown(employeeId), 
  }; 
} 
 
async disbursePay(employeeId: string, amount: number, processedById: string, note?: string) { 
  const summary = await this.getEmployeeBalanceSummary(employeeId); 
  if (amount > summary.currentBalance) { 
    throw new UnprocessableEntityException({ 
      code: 'PAYMENT_EXCEEDS_BALANCE', 
      message: `Cannot disburse ${formatPKR(amount)}. Balance is 
${formatPKR(summary.currentBalance)}`, 
    }); 
  } 
  return this.prisma.payment.create({ 
    data: { employeeId, amount, processedById, note }, 
  }); 
}
